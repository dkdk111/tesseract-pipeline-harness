# Examples: a gallery across domains and perspectives

Six demos, each from a different domain and each with a different structural
signature, so the harness is convincing from more than one angle. Every one runs
with no model and no keys. See them all at once:

    python -m tesseract_pipeline gallery

```
example                domain         O B D T  leaf  perspective
01_market_brief        research       O B D T   10   all four axes open (a full tesseract)
02_software_release    devops         O B . .    6   order + breadth, with a human approval gate on deploy
03_data_pipeline       data / etl     O B D .    8   order, breadth, and depth, with no time axis
04_book_chapter        writing        O B . T    5   time wrapping order and breadth (a revision loop)
05_bulk_translation    localization   . B . .    6   breadth only, and it hits the box's max_breadth wall
06_quick_fix           maintenance    . . . .    1   a single leaf: the harness refuses to invent dimensions
```

(O = order, B = breadth, D = depth, T = time; a dot means the axis was not opened.)

The point of the spread is that the harness opens exactly the axes each job needs,
and no more. A pipeline is a tesseract only when the work is one; when the work is a
line, an honest self-design leaves it a line.

## What each demo shows

- 01_market_brief (research). The flagship: one task opens all four axes. Time (a
  revision loop) wraps order (collect, analyze, write, review) wraps breadth
  (parallel competitors and lenses) wraps depth (an oversized competitor becomes a
  seed).
- 02_software_release (devops). A familiar CI/CD shape: build, then parallel test
  suites, then package, then deploy. Deploy is high-risk, so it is held at the
  human approval gate, never self-designed or executed autonomously. This is the
  control perspective.
- 03_data_pipeline (data / etl). An ETL job: parallel ingestion, then validation,
  then a transform that is too large to handle flat and opens depth, then load. It
  opens depth but not time, showing the harness does not force a revision loop where
  the work does not need one.
- 04_book_chapter (writing). The self-example, close to how this book was written:
  a draft-review-revise loop (time) over an outline-draft-edit spine (order), with
  the sections drafted in parallel (breadth).
- 05_bulk_translation (localization). Embarrassingly parallel: one document into
  eight languages, all independent. Only breadth opens. It also declares more
  branches than the box allows, so it hits the max_breadth wall and the surplus is
  queued, not dropped silently. This is the box-wall perspective.
- 06_quick_fix (maintenance). A one-line job. No axis opens; it stays a single
  leaf. This is the restraint perspective: opening dimensions the work does not need
  is a self-design failure, not thoroughness.

## Each demo's files

- `task.json`: the input. It declares the work's nature (and, for the gallery, a
  `domain` and `perspective` label the planner ignores). It never names an axis.
- `tesseract.json`, `trace.md`, `output.md`: generated by the engine and committed
  so you can browse them here without running anything.

## Run or regenerate one

    python -m tesseract_pipeline render examples/02_software_release/tesseract.json
    python -m tesseract_pipeline run    examples/02_software_release/task.json --out examples/02_software_release

## Make the leaves real

Swap the deterministic simulator for a model. Only the worker changes; the planner,
the executor, and the box stay identical. See `llm_worker_example.py`.
